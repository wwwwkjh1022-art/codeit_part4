# 소상공인 AI 홍보 자동화 서비스 업무일지 정리

## 프로젝트 개요

- 프로젝트명: 소상공인 AI 홍보 자동화 서비스
- 개발 형태: FastAPI 기반 MVP 구현
- 목표: 소상공인이 간단한 브리프만 입력해도 Instagram, Threads, Blog용 홍보 콘텐츠와 배너 이미지를 생성하고, 결과 화면에서 수정 후 게시 준비까지 할 수 있는 흐름을 만드는 것

## 작업 기간

- 정리 기준 작업 기간: 2026-04-13 ~ 2026-04-27
- 진행 방식: 초기 기획/조사, 구조 설계, 기능 구현, UI 개선, 생성 품질 보강, 게시 연동, 문서화 병행

### 기간 정리 기준

- 2026-04-13 ~ 2026-04-22: 문제 정의, 기능 기획, 채널별 요구사항 조사, 서비스 구조 구상
- 2026-04-23 ~ 2026-04-27: 실제 구현, 테스트, UI 개선, 문서화, 제출 자료 정리

즉, 앞부분은 "무엇을 만들지 정리한 기간", 뒷부분은 "실제로 코드와 결과물을 만든 기간"으로 구분해 정리했다.

## 프로젝트 전반에서 신경 쓴 부분

### 1. 생성만 하고 끝나지 않게 만들기

- 단순히 문구를 한 번 생성하는 것이 아니라
- 이미지 재생성
- 채널별 미리보기
- 문구 수정
- 계정 연결
- 예약/게시 준비

까지 이어지는 흐름을 만들려고 했다.

### 2. 외부 API 실패를 서비스 실패로 만들지 않기

- OpenAI 실패 시 mock fallback
- SD 계열 이미지 생성 실패 시 local fallback
- 네이버 블로그 연결 실패 시 수동 완료 fallback

처럼, "전체 흐름이 멈추지 않는 구조"를 중요하게 봤다.

### 3. 발표와 제출까지 고려한 프로젝트로 만들기

- README, API 문서, 자동화 문서, 포트폴리오 타임라인, 발표 자료를 함께 정리했다.
- 단순히 코드만 있는 저장소가 아니라, 설명 가능한 프로젝트 형태로 마무리하려고 했다.

### 4. 기획 기간과 구현 기간을 분리해서 보기

- 앞부분은 문제 정의와 구조 설계에 시간을 썼고
- 뒷부분은 실제 구현과 검증에 집중했다.
- 이 구분 덕분에 발표할 때도 "왜 이런 구조로 만들었는지"를 설명하기 쉬워졌다.

## 최종 소감

이번 작업은 단순히 AI API를 붙이는 수준이 아니라, 실제로 사용할 수 있는 서비스 흐름을 설계하는 경험에 가까웠다. 기능 하나를 구현할 때마다 "이 결과를 사용자가 다음에 어떻게 쓸까"를 같이 고민해야 했고, 그 과정에서 생성, 저장, 수정, 게시 준비, 문서화까지 전체 서비스를 보는 시야가 중요하다는 것을 배웠다.

특히 fallback 구조, 결과 페이지 UX, 채널별 게시 전략 분리, 제출 문서 정리를 통해 단순 데모를 넘어 "설명 가능한 서비스형 MVP"로 프로젝트를 마무리할 수 있었다.
