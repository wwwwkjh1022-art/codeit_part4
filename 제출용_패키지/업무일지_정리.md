# 소상공인 AI 홍보 자동화 서비스 업무일지 정리

## 프로젝트 목표

소상공인이 최소한의 정보만 입력해도 광고 문구와 채널별 홍보 콘텐츠를 빠르게 생성할 수 있는 서비스를 구현하고, 향후 자동 예약 게시까지 확장 가능한 구조를 만드는 것을 목표로 작업했다.

## 업무일지

### 1. 프로젝트 초기 세팅

- Python 가상환경과 기본 디렉터리 구조를 구성했다.
- `app`, `routes`, `services`, `schemas`, `templates`, `static`, `tests` 구조를 나누어 시작했다.
- `.env`, `.gitignore`, 실행 구조를 정리했다.

배운 점:

- 처음부터 구조를 나눠두면 이후 기능 확장과 테스트 작성이 훨씬 쉬워진다.

### 2. FastAPI 기반 MVP 구성

- 입력 페이지와 결과 페이지를 만들었다.
- `GET /`, `POST /generate`, `GET /health` 라우트를 구성했다.
- 사용자가 브리프를 입력하고 결과를 바로 볼 수 있는 기본 흐름을 완성했다.

고민한 점:

- 빠르게 만들 수 있는 Streamlit도 고려했지만, 장기적으로 서비스 구조를 보여주기에는 FastAPI가 더 적합하다고 판단했다.

### 3. 생성 로직 분리

- AI 생성기를 추상화해 provider 교체가 가능한 구조로 분리했다.
- OpenAI API를 사용할 수 있도록 구현했다.
- API 실패 시 fallback으로 mock 생성기가 작동하도록 설계했다.

배운 점:

- 외부 API 의존 기능은 예외 상황을 먼저 고려해야 서비스 전체가 안정적이다.

### 4. 결과 구조 고도화

- 단일 광고 문구가 아니라 Instagram, Threads, Blog 패키지로 확장했다.
- 각 채널별로 필요한 필드가 다르기 때문에 결과 스키마를 세분화했다.
- CTA, 추천 시간, SEO 키워드, visual hook 등 실사용 요소를 추가했다.

아쉬운 점:

- 처음에는 단순 캡션 중심 구조여서 확장성이 부족했다.
- 중간에 결과 스키마를 크게 바꾸면서 테스트도 함께 수정해야 했다.

### 5. 배너 미리보기 구현

- Pillow로 배너 이미지를 생성하도록 구현했다.
- 업로드 이미지가 있을 때와 없을 때를 모두 처리했다.
- 스타일 옵션별 색감 차이를 주어 결과물이 더 서비스답게 보이도록 했다.

배운 점:

- 텍스트 생성만 보여주는 것보다 시각 결과물이 있으면 서비스 완성도가 훨씬 높아진다.

### 6. 입력 검증 및 오류 처리

- 입력값 검증 실패 시 에러가 500으로 터지지 않도록 예외 처리를 추가했다.
- 사용자에게 어떤 항목을 수정해야 하는지 한국어 메시지로 안내했다.
- 상품 설명을 비워도 자동 설명을 채우도록 개선했다.

좋았던 점:

- 사용성 관점에서 “입력이 조금 부족해도 서비스가 동작한다”는 느낌을 줄 수 있었다.

### 7. 품질 평가 구조 설계

- 생성 결과가 실제로 좋은지 평가하기 위한 기준을 만들었다.
- Hook, Specificity, Channel-Native Readiness, CTA, Compliance 같은 항목을 점수화했다.
- 테스트셋과 평가 리포트를 별도 파일로 관리했다.

배운 점:

- AI 결과물은 단순히 생성되었다는 것만으로 충분하지 않고, 평가 기준을 함께 설계해야 설득력이 생긴다.

### 8. 생성 이력과 재생성 기능 추가

- 생성 결과를 로컬 저장소에 저장하도록 했다.
- 과거 캠페인을 다시 열어볼 수 있게 했다.
- 같은 브리프로 재생성하는 흐름을 넣었다.

의미:

- 단순 데모를 넘어서 서비스처럼 사용할 수 있는 기본 흐름이 생겼다.

### 9. 예약 및 자동화 준비

- 게시 예약 작업을 저장할 수 있게 했다.
- mock 게시 adapter를 구현해 실제 연동 전 흐름을 검증할 수 있게 했다.
- n8n이 사용할 API와 workflow 파일을 정리했다.

고민한 점:

- 실제 계정/비밀번호 방식은 위험하다고 판단해 공식 API/OAuth 방향만 남기고 mock으로 제한했다.

### 10. 제출용 문서와 발표 대응 자료 정리

- README를 정리해 실행 방법, 환경 변수, 기능, 테스트, 문서 위치를 확인할 수 있게 했다.
- 평가 리소스와 자동화 문서를 별도로 정리했다.
- 프로젝트 개발 흐름을 포트폴리오 타임라인 문서로 남겼다.

## 프로젝트 진행 중 느낀 점

- 이번 프로젝트는 단순히 AI 호출 한 번으로 끝나는 과제가 아니라, 결과를 실제 서비스처럼 보여주고 검수 흐름까지 만드는 것이 중요했다.
- 기능을 추가할수록 “실제로 이걸 제출 때 어떻게 설명할지”를 같이 고민해야 했다.
- mock fallback, 품질 점수, 배너 미리보기, 이력 저장 같은 요소들이 전체 완성도를 크게 높여줬다.

## 최종 정리

이번 프로젝트를 통해 FastAPI 기반 서비스 구조 설계, AI 응답 스키마 설계, 예외 처리, 결과 평가 지표 설계, 문서화, 자동화 확장 설계까지 한 번에 경험할 수 있었다. 제출용 결과물 기준으로는 핵심 요구사항을 충족하는 수준의 MVP를 완성했다고 판단한다.
