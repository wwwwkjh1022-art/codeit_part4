  • 로컬에서 빠르게 검증 가능한 MVP 범위를 우선 정리했다.

4. 프로젝트 전반에서 신경 쓴 부분

  • 단순 생성이 아니라 이미지 재생성, 채널별 미리보기, 문구 수정, 계정 연결, 예약/게시까지 이어지는 흐름을 만들려고 했다.
  • OpenAI 실패 시 mock fallback, SD 실패 시 local fallback, 네이버 블로그 연결 실패 시 수동 완료 fallback처럼 전체 흐름이 끊기지 않도록 구성했다.
  • README, API 문서, 자동화 문서, 포트폴리오 타임라인, 발표 자료를 함께 정리해 설명 가능한 프로젝트 형태로 마무리했다.

6. 최종 소감

이번 작업은 단순히 AI API를 붙이는 수준이 아니라, 실제로 사용할 수 있는 서비스 흐름을 설계하는 경험에 가까웠다. 기능 하나를 구현할 때마다 "이 결과를 사용자가 다음에 어떻게 쓸까"를 함께 고민해야 했고, 그 과정에서 생성, 저장, 수정, 게시 준비, 문서화까지 전체 서비스를 보는 시야가 중요하다는 점을 배울 수 있었다.

특히 fallback 구조, 결과 페이지 UX, 채널별 게시 전략 분리, 제출 문서 정리를 통해 단순 데모를 넘어 설명 가능한 서비스형 MVP로 프로젝트를 마무리할 수 있었다.
